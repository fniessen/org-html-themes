This document demonstrates a broad range of Org mode features. It serves as
a comprehensive reference for writing technical documentation, producing HTML
and PDF output, embedding source code, and testing CSS, LaTeX, and export
behavior.

*Org mode* provides a simple /plain-text/ markup language for creating
well-structured documents. From a single source file, you can produce
high-quality HTML, LaTeX, PDF, and many other output formats while maintaining
an efficient writing workflow.

#+end_export

* Basics

This chapter introduces the fundamental building blocks of an Org document.

** Biggest heading

Start a new chapter by creating a top-level heading.

*** Bigger heading

Use section headings to organize related content.

**** Big heading

Subsections help divide longer sections into manageable parts.

**** Paragraphs and line breaks

A single newline does not create a new paragraph.
The following lines belong to the same paragraph.

But an empty line

demarcates a new paragraph.

To force a line break without beginning a new paragraph, end a line with
2 backslashes. \\
The next line will appear immediately below it.

To insert a horizontal rule, write at least 5 consecutive hyphens: this text
appears above the horizontal rule.
-----
and this text appears below it.

**** Numbered headings

You can automatically number headings up to a specified depth by setting the
appropriate export option.

#+begin_src org
,#+OPTIONS: H:4
#+end_src

One morning, when Gregor Samsa woke from troubled dreams, he found himself

** Lists

Org mode provides several types of lists that can be freely combined to organize
information clearly and consistently.

*** Unordered lists

Bulleted lists are useful for presenting related information without implying
a particular order.

They are commonly used to:

- organize information, and
- present content
  + hierarchically, and
  + easy to read.

You can begin list items with either a minus (~-~) or a plus sign (~+~).

*** Ordered lists

Numbered lists express sequences or ordered procedures.

1. First item
   1) First sub-item
   2) Second sub-item
2. Second item

You can also specify explicit numbering when necessary:

1. [@1] First item
2. [@2] Second item
5. [@5] Continue with item 5

*** Definition lists

Definition lists associate terms with descriptions.

- Definition list ::
     A list in which each item consists of a term followed by its definition.

- Org mode ::
     A powerful plain-text authoring system built into Emacs.

*** Checklists

Checkboxes make it easy to track progress.

- [ ] Incomplete task
- [-] Task in progress
  + [X] First subtask
  + [ ] Second subtask
- [X] Completed task

** Additional features

*** Including other Org files

You can include the contents of another Org file during export. For example, the
following directive imports the referenced file while skipping its title:

#+begin_src org
,#+INCLUDE: chapter1.org :lines "2-"
#+end_src

#+begin_note
File inclusion using the ~#+INCLUDE~ keyword is supported *during export only*.
#+end_note

*** Inline HTML

Raw HTML can be embedded directly in an Org document. When exporting to HTML,
the markup is preserved exactly as written.

#+HTML: <pre style="color:red">
This text appears in a fixed-width font.
#+HTML: </pre>

Embedding raw HTML is particularly useful when you need functionality beyond Org
mode's built-in export options, such as custom markup, advanced styling, or
interactive content.

For example, you can define custom HTML containers that correspond to CSS
classes:

#+begin_example
,#+begin_info
,*Info example* \\
Additional details appear here.
,#+end_info
#+end_example

You may also embed JavaScript, interactive visualizations (such as interactive
R plots), or other HTML elements when appropriate.

To include an existing HTML file verbatim during export:

#+begin_src org
,#+INCLUDE: file.html html
#+end_src

Changes should always be made to the original Org source rather than the
exported HTML output!

*** Inline LaTeX

Raw LaTeX may also be embedded directly in an Org document.

#+LATEX: \begin{verbatim}
This text is displayed using a fixed-width font.
#+LATEX: \end{verbatim}

*** Centered text

Use a ~center~ block to center its contents.

#+begin_center
This text is centered.
#+end_center

** Code blocks

Source code blocks are one of Org mode's most powerful features. They support
syntax highlighting, executable code through Org Babel, export to multiple
formats, and *reproducible research* workflows.

*** COMMENT Syntax highlighting

The following example demonstrates syntax highlighting for a C++ source block.

#+begin_src cpp :eval no
/*
 * Simple application that writes a greeting to standard output.
 */
int main(int argc, char **argv)
{
  printf("Hello, %s!\n", (argc > 1) ? argv[1] : "World");
  return 0;
}
#+end_src

Org Babel currently recognizes many programming languages.

#+begin_src emacs-lisp :results drawer :exports results
(concat (mapconcat (lambda (widget)
                     (widget-get widget :tag))
                   (cl-remove-if-not (lambda (it)
                                       (and (consp it)
                                            (eq (car it) 'const)))
                                     (cdr
                                      (widget-get
                                       (get
                                        'org-babel-load-languages 'custom-type)
                                       :key-type)))
                   ", ")
        ".")
#+end_src

*** Line numbers

Both ~example~ and ~src~ blocks support optional line numbering. Add the ~-n~
switch to the ~#+begin~ line to display line numbers.

#+header: :eval no
#+begin_src emacs-lisp -n
(defun org-xor (a b)
  "Exclusive or."
#+end_src

To continue numbering from a previous numbered block, use the ~+n~ switch.

#+header: :eval no
#+begin_src emacs-lisp +n
  (if a (not b) b))
#+end_src

Literal example blocks can also define labels such as ~(ref:name)~. These labels
become reference targets that may be linked elsewhere in the document, using
special hyperlinks like ~[[(name)]]~ (i.e., with the reference name enclosed in
single parenthesis).

(In HTML, hovering the mouse over such a link will remote-highlight the
corresponding code line, which is kind of cool.)

Use the ~-r~ switch to suppress labels in the exported source while still
allowing references.

With the ~-n~ switch, links to these references will be labeled by the line
numbers from the code listing, otherwise links will use the labels with no
parentheses.

#+header: :eval no
#+begin_src emacs-lisp -n -r
  (save-excursion                  ; (ref:sc)
    (goto-char (point-min)))       ; (ref:jump)
#+end_src

In line [[(sc)]], the current cursor position is saved. [[(jump)][Line (jump)]] moves directly
to ~point-min~.

*** Program output

One of Org mode's most powerful features is the ability to *execute* code and
insert its output directly into the document. This enables /reproducible
research/.

**** Text output

A simple example:

#+begin_src shell :exports both :results verbatim
date +"%Y-%m-%d"
#+end_src

#+results:
: 2026-07-15

Longer command output may also be inserted into the document.

#+name: list-of-styles
#+begin_src shell :exports both :results verbatim
# Output all styles, but the default one (if any).
ls ../src/ | grep -v "default"
#+end_src

#+results: list-of-styles
#+begin_example
bigblow
readtheorg
vendor
#+end_example

Org mode supports several forms of inline emphasis.

Examples include:

- /italic/
- *bold*
- */bold italic/*
- _underlined_
- ~inline code~
- =verbatim text=
- +deleted text+

*** Quotations

Use a ~quote~ block for quoted prose.

#+begin_quote
Let us change our traditional attitude to the construction of programs:
Instead of imagining that our main task is to instruct a computer what to do,
let us concentrate rather on explaining to human beings what we want a
computer to do.

The practitioner of literate programming can be regarded as an essayist, whose
main concern is with exposition and excellence of style. Such an author, with
thesaurus in hand, chooses the names of variables carefully and explains what
each variable means. He or she strives for a program that is comprehensible
because its concepts have been introduced in an order that is best for human
understanding, using a mixture of formal and informal methods that reinforce

--- Donald Knuth

Short quotations are equally straightforward.

#+begin_quote
Everything should be made as simple as possible,

#+begin_verse
Everything should be made as simple as possible,

** Mathematical expressions

Org mode supports LaTeX mathematics in both HTML (via MathJax) and LaTeX/PDF
exports.

- For *inline mathematics*, use ~\(...\)~: \(x^2\) or \(1 < 2\).

  Although ~$...$~ is commonly used in LaTeX, ~\(...\)~ is preferred in Org
  documents.

- Display equations appear centered on their own line (the /Euler theorem/):

  \[
  \int_0^\infty e^{-x^2} dx = \frac{\sqrt{\pi}}{2}
  \]

  Display math using ~\[...\]~ is useful whenever an equation deserves *visual
  emphasis*.

  For example, the /quadratic formula/ may be written as:
  \[ \frac{-b \pm \sqrt{b^2 - 4ac}}{2a} \]

  *Double dollar signs (~$$~) should not be used*.

- For *numbered* equations, use the ~equation~ environment. The /sinus theorem/ can
  then be written as the equation:

  \begin{equation}
    \label{eqn:sinalpha}
    \frac{\sin\alpha}{a}=\frac{\sin\beta}{b}
  \end{equation}

- Equation references are also supported.

  See Equation [[the-first]].

  #+name: the-first
  \begin{equation}
    n_{i+1}= \frac{n_{i}(d-i)(e-1)}{(i+1)}
  \end{equation}

  Only captioned equations receive automatic numbering.

- If numbering is not required, use ~equation*~, ~displaymath~, or the shorter
  ~\[...\]~ syntax.

# Differently from $...$ and \(...\), an equation environment produces a *numbered*
# equation to which you can add a label and reference the equation by (label)
# name in other parts of the text. This is not possibly with unnumbered math
# environments ($$, ...).

** Tables

Org mode makes it easy to create and maintain tables.

#+CAPTION: Example table
| Header 1    | Header 2      | Header 3 |
|-------------+---------------+----------|
| Top left    | Top middle    |          |
|             |               | Right    |
| Bottom left | Bottom middle |          |

Columns are aligned automatically.

- Numeric columns are right-aligned.
- Text columns are left-aligned.

Alignment can also be specified explicitly.

#+CAPTION: Table with explicit alignment
|          <r> |     <c>      | <l>          |
|            1 |      2       | 3            |
|        right |    center    | left         |
| xxxxxxxxxxxx | xxxxxxxxxxxx | xxxxxxxxxxxx |

Placement may be controlled during LaTeX export.

#+ATTR_LATEX: :center nil
| a | b |
|---+---|
| 1 | 2 |

This differs from the default centered table.

| a | b |
|---+---|
| 1 | 2 |

*** Table alignment on the page

Left-aligned table:

#+LATEX: \noindent
#+ATTR_LATEX: :center nil
| a | b | c |
|---+---+---|
| 1 | 2 | 3 |
| 4 | 5 | 6 |
#+LATEX: \hfill

The ~noindent~ just gets rid of the indentation of the first line of a paragraph
which in this case is the table, and the ~hfill~ adds infinite stretch after the
table, so it pushes the table to the left.

Centered table:

| a | b | c |
|---+---+---|
| 1 | 2 | 3 |
| 4 | 5 | 6 |

Right-aligned table:

#+LATEX: \hfill
#+ATTR_LATEX: :center nil
| a | b | c |
|---+---+---|
| 1 | 2 | 3 |
| 4 | 5 | 6 |

Here, the ~hfill~ adds infinite stretch before the table, so it pushes the table
to the right.

#+begin_info
*Info example* \\
Did you know...
#+end_info

*** Tip

A tip box is displayed as follows:

#+begin_tip
*Tip example* \\
Consider using this approach...
#+end_tip

*** Note

A note box is displayed as follows:

#+begin_note
*Note example* \\
This information may be useful.
#+end_note

*** Warning

A warning box is displayed as follows:

#+begin_warning
*Warning example* \\
Verify your configuration before proceeding.
#+end_warning

Org mode supports several types of links.

Most links point to headings, but they may also reference custom anchors within
the current document or another Org document.

You can create an anchor using ~<<name-of-anchor-here>>~ <<name-of-anchor-here>>.

* Org mode features

** Dates

Org mode recognizes both inactive and active timestamps.

Inactive timestamp: [2014-01-16 Thu]

Active timestamp: <2014-01-16 Thu>

Completed tasks are typically collapsed by default in exported documents.

Scheduled tasks remain expanded unless configured otherwise.

To collapse all headings when exporting HTML with BigBlow, include:

#+begin_src org
,#+HTML_HEAD: <script> var HS_STARTUP_FOLDED = true; </script>
#+end_src

into your Org document.

** Personal note                                              :computer:write:

Tags may be assigned to headings to improve organization and searching.

Interactive HTML exports (such as BigBlow) can also highlight entries that share
the same tag.

** Weekly review                                               :computer:

The BigBlow HTML export supports a "review mode"...

Press the ~r~ key to collapse every active heading except the current one, making
it easier to focus on a single task during a review session!

* Org macros

Macros allow reusable snippets to be defined once and expanded throughout the
document.

#+MACRO: color @@html:<span style="color: $1">$2</span>@@ @@latex:\textcolor{$1}{$2}@@ @@odt:<text:span text:style-name="$1">$2</text:span>@@

Example:

{{{color(blue, This text is displayed in blue.)}}}

{{{color(red, This other text is displayed in red.)}}}

Additional macros are available on [[https://github.com/fniessen/org-macros][GitHub]].

* BigBlow extensions

Some HTML themes provide additional features.

For example, the keyword ~FIXME~ may automatically be replaced with a visual
"Fix Me!" indicator:

FIXME Delete this...
